# Step 26 — World / Structure editor

**Phase I · needs step 25**

## Goal

Author maps and prefabs visually: place/erase/orient voxels, box fills, markers, portals,
trainers, prefab stamping — replacing hand-written map JSON.

## Reading

[tooling.md](../../03-systems/tooling.md) ·
[02-data-model.md §13/§13b](../../02-data-model.md) · `MousePicker`/`WorldRaycaster`
(step 07 — reused directly).

## Files

`tools/pages/world_editor_page.hpp/.cpp` — center: Viewport3D over an editable
`VoxelWorld` (chunk sync pipeline from step 06 gives free live remeshing); left: voxel
palette bar (registry list + search) + tool modes (place, erase, box-fill, pick-orient
[R rotates, F flips], marker, portal, trainer, stamp); right: PropertyPanel for the
selected element (marker name, portal target map/portal IdPickers, trainer fields, map
metadata: size, biome IdPicker); mouse: MousePicker against the edit world (place targets
the DDA hit's adjacent cell, erase the hit cell); New Map / New Prefab (prefabs edit the
same way minus map-only fields, plus anchor placement); Save → `maps/` / `prefabs/` via
JsonWriter (fills are **not** reverse-engineered — the editor keeps authored fills if the
file had them, else writes dense sparse cells; note it in the file header comment…
JSON has no comments — in the doc instead: cells-only output is acceptable).
`tools/core/edit_camera` — fly/orbit camera preset for editing (reuses Viewport3D input,
adds pan).

## Tests (`[test]`)

Map/prefab round-trip through the writer incl. stamps/portals/trainers/anchors; box-fill
cell math; place-adjacent resolution (DDA face → neighbour cell).

## Definition of Done

`[tools]` (user validates): rebuild the M1 testground *in the editor* (or edit it), add a
prefab house + stamp it, wire a portal pair, save; `[run]` the game on the edited map —
everything works (portals, trainer, bushes). The `m1-testground.json` in the repo is
regenerated by the editor from this step on.
